# システム構築の要件定義

- 文書名: システム構築の要件定義
- 対象: Kabu Station API を利用した日本株自動売買システム
- 版数: v3.0 (AI統合アップデート版)
- 前提市場データ: J-Quants Standard プラン
- 作成方針: 本番環境の安全性・再現性を担保しつつ、AI・ニュース分析による高度なクオンツ運用アーキテクチャを実現する

---

## 1. 背景と目的

三菱UFJ eスマート証券の **kabuステーション API** を用いて、日本株の自動売買環境を構築する。
本システムは、単なる発注自動化ではなく、以下を満たすことを目的とする。

- スクリーニングにより売買候補銘柄を抽出できること（データドリブン）
- 複数ファクターとAIニュース分析・AIレジーム判定を統合した売買判断ができること
- バックテストにより売買戦略を事前検証し、本番環境と一致する再現性を担保できること
- 本番売買において誤発注防止・損失制御・監査証跡を確保できること
- 研究、検証、本番の各環境を分離し、段階的に高度化（AI予測モデル等）へ拡張できること

---

## 2. システム化の目的

### 2.1 業務目的

- 日本株の売買意思決定から発注までを自動化し、手作業負荷を軽減する
- 売買ルールをコード化し、感情を排した再現可能な運用を実現する
- 単一指標ではなく、モメンタム・バリュー・流動性に加え、ニュース・マクロAI分析を取り入れた多角的な判断基盤を構築する

### 2.2 技術目的

- kabuステーション API を利用したミリ秒単位の安定した売買執行基盤を整備する
- J-Quants Standard の市場データを主とする日次分析基盤を構築する
- 戦略（AI・スコアリング）ロジックと執行ロジックを分離し、拡張性を高める
- 取引ログ、スコア判定根拠、バックテスト条件を確実に追跡可能にする

---

## 3. 適用範囲と前提条件

### 3.1 取引対象

- 東京証券取引所上場の現物株（プライム・スタンダード市場中心）
- 売買単位は 100 株単位を基本とする
- 日次スクリーニングと日中自動売買を対象とする

### 3.2 Kabu Station API に関する前提（重要制約）

- 利用には **Professional プランまたは Premium プラン** 条件を満たす必要がある
- kabuステーションが起動している同一PC / 同一IP からの利用を前提とする
- PUSH 配信（WebSocket）の登録可能銘柄数は **最大50銘柄** とする
- API リクエスト上限: 発注系 **5件/秒**、情報系・余力系 **10件/秒** を厳守する
- 夜間メンテナンス帯は自動売買プロセスを停止する

### 3.3 J-Quants に関する前提

- データ取得元は **J-Quants Standard プラン** とする（日足、銘柄情報、財務情報）
- API 利用上限は **120 リクエスト/分** を前提とし、スロットリングを実装する

---

## 4. システム全体構成（5層アーキテクチャ）

研究環境から本番環境までを一気通貫で捉える5層構造を採用する。

1. **Research Layer (研究層)**
   - J-Quants データ取込 / 特徴量生成 / スクリーニング
   - バックテスト / IN-SAMPLE検証
2. **AI Analysis Layer (AI分析層)**
   - ニュースセンチメント分析 (NLP)
   - 市場レジーム判定 (Bull/Neutral/Bear)
3. **Strategy Layer (戦略層)**
   - モメンタム、バリュー等とAIスコアの統合（シグナル生成）
4. **Execution Layer (執行層)**
   - kabuステーションAPI (REST/WebSocket) との通信
   - 注文生成 / リスク管理（上限チェック等） / 約定管理
5. **Monitoring Layer (監視層)**
   - ポジション管理 / ログ・監査証跡 / エラー・アラート通知

## 5. 機能要件

### 5.1 市場データ取得・保存機能 (ETL)

- J-Quants API等から日次でデータを取得し、RDB（PostgreSQL等）へ保存する。
- 保存対象：株価履歴 (OHLCV)、財務データ、コーポレートアクション情報、銘柄マスタ。
- 夜間バッチで実行し、エラー時はリトライ・通知を行う。欠損値の補完や異常値検知を包含する。

### 5.2 特徴量・AI分析機能

- **テクニカル/財務特徴量**: リターン、移動平均乖離、ボラティリティ、PER、PBR等の生成。
- **AIニュース分析**: Yahoo!ニュース・日経等からNLPでポジ/ネガ/中立を判定し、`news_score` を算出。
- **市場レジーム判定**: マクロ指標や全体のボラティリティから相場状態（Bull/Neutral/Bear）を判定し、リスク許容度を決定。

### 5.3 スクリーニング・シグナル生成機能

- 上記特徴量とAIスコアを重み付け統合し、`final_score` を算出する。
- 設定した閾値を越える銘柄から、売買候補リストを生成（買い・売り・見送り）。
- シグナルとその生成根拠（各スコア詳細）は必ずDBへ保存する。

### 5.4 バックテスト機能

- 戦略を過去データ（最低5年推奨）で日足ベースで検証可能にする。
- **Look-ahead bias（未来事象の参照）** を防止する仕組みを必須とする。
- 売買手数料・スリッページを反映した上で、CAGR、Sharpe Ratio、Max Drawdown等を算出する。

### 5.5 発注・約定管理機能 (kabuステーション接続)

- 決定したシグナルに基づき、kabuステーション API へ発注リクエスト（成行・指値等）を送信。
- 注文前に余力、価格、数量、時間帯、APIレート制限をチェックする。
- 注文状態（約定・部分約定・失効）を追跡し、ローカルのポジション情報を更新する。
- 二重発注防止のための冪等（等価）キー処理を実装する。

### 5.6 リスク・ポジション管理機能

- **ポジション管理**: 1銘柄の最大投資額（例:総資産の10%上限）、最大保有数、セクター上限の制御。
- **損失管理**: 最大ドローダウン超過時や連続損失上限に達した場合の自動新規発注停止。
- **障害時制御**: APIトークン取得失敗、WebSocket切断遅延、PC時刻同期異常時はシステムを安全に停止。手動キルスイッチ（即時停止）を用意する。

## 6. 非機能要件

### 6.1 可用性・性能

- 営業時間内（日中）は常駐稼働し、プロセス死活監視を行う。異常終了・再起動後は自動復旧（状態復元）が可能であること。
- シグナル判定から発注までの処理遅延を許容範囲内に抑え、APIレート制限（5件/秒 等）を厳守するキュー機構を持つこと。

### 6.2 監査・ログ記録機能

- 再現性確保のため、すべてのAPI送受信、シグナル判定に使用した特徴量・閾値、環境変数バージョンを記録する。
- 日次で損益レポートやエラー履歴を出力可能とする。

### 6.3 セキュリティ

- kabuステーション API パスワード、トークンなどは環境変数やシークレット管理マネージャで保護し、平文ログに残さないこと。

---

## 7. 開発・導入フェーズ

初期フェーズから安全を最優先とし、段階的にAI要素を追加する。

- **Phase 1: PoC（ベースライン）**: J-Quants 取込、基本特徴量スクリーニング、単純バックテスト、kabu API 疎通確認。
- **Phase 2: 検証・バックテスト基盤**: 特徴量DB整備、複数戦略（モメンタム・バリュー）のバックテスト、ログ・監視の整備。
- **Phase 3: 小額本番 (ルールベース)**: 本番接続、監視対象絞込、損失上限・キルスイッチを稼働させた少額運用開始。
- **Phase 4: AI機能の統合 (本番環境)**: AIニュース分析 (NLP) とレジーム判定モデルを稼働させ、スコアリングを統合。
- **Phase 5: 高度化**: 完全なAI予測モデルへの移行、ポートフォリオ動的最適化。
