# 個人が作れる最強の日本株自動売買アーキテクチャ

## 1. 目的

本ドキュメントは、**J-Quants Standard** を分析・研究基盤、**kabuステーション API** を本番執行基盤として用い、個人投資家が実運用可能な中で最もプロ仕様に近い、**AI判断モデル（ニュース・マクロレジーム）を統合した日本株自動売買アーキテクチャ** を定義するものである。

目標は次の4点とする。

- スクリーニング、バックテスト、実売買を一貫した設計でつなぐ（データドリブン）
- 誤発注や過大リスクを防ぐ安全装置（キルスイッチ・3段階ガード）を標準装備する
- 研究環境（Linux/Cloud等）と本番執行環境（Windows）を物理・論理的に分離し、分析負荷による注文遅延を防ぐ
- 個人開発の現実的な制約の中で、AIによる「情勢補正」を取り入れた最強のクオンツ構成を実現する

---

## 2. 設計方針とアーキテクチャ5層構造

最強構成といっても、1台のPCにすべてを詰め込むことを目指さない。重要なのは、以下の5層を**論理的・物理的に分離**することである。

1. **Research Layer (研究・分析系)**：J-Quantsからのデータ取得、特徴量生成、バックテスト
2. **AI Analysis Layer (AI判断モデル層)**：ニュース・マクロ情報のNLP解析、レジーム判定
3. **Strategy Layer (戦略運用系)**：複合スコア（財務・テクニカル＋AI）によるシグナル生成、ポジション制御
4. **Execution Layer (執行系)**：kabu APIを通じた注文送信、約定確認、再送、緊急停止
5. **Monitoring Layer (運用監視系)**：状態監視、ログ、アラート、日次レポート

### 採用前提スタック

- **市場データ**：J-Quants Standard
- **注文執行**：kabuステーション API
- **実行OS**：執行系はWindows（kabuステーション稼働必須）、研究・AI系は任意（Linux推奨）
- **開発言語**：Python 3.11+ (pandas, polars, scikit-learn, LLM/NLPライブラリ)
- **保管DB**：PostgreSQL
- **システム間連携**：Redis（キャッシュ/キュー）
- **監視通知**：Slack / Discord / Email

## 3. 推奨配置構成（物理/論理配置）

個人運用における「最強構成」は、以下のように**分析PCと執行PCを分ける**ことである。

```mermaid
flowchart LR
    subgraph PC2[分析・AIサーバ / Linux or Cloud]
        J1[J-Quants ETL & Data Store]
        A1[News Collector & NLP Analyzer]
        F1[Feature / Screening Engine]
        B1[Backtest Engine]
        R1[(Research & AI DB)]

        J1 --> R1
        A1 --> R1
        F1 --> R1
        B1 --> R1
    end

    subgraph PC1[本番執行PC / Windows限定]
        K1[kabuステーション API]
        E1[Execution App (Order/Position Manager)]
        S1[Strategy & Risk Manager]
        D1[(PostgreSQL / Redis)]
        M1[Monitoring Agent]

        K1 --- E1
        E1 --- S1
        S1 --- D1
        E1 --- M1
    end

    PC2 -. 翌日ユニバース・AIシグナル配信 .-> PC1
    PC1 -. 実行結果/監査ログ返却 .-> PC2
```

### なぜ分けるか？

- NLPや特徴量生成の重い処理で、Windows(執行PC)のCPUを専有し、kabuステーションをフリーズさせないため。
- 誤ってバックテストコードから本番のREST APIへ注文を出さないため。
- Windows必須という「証券会社の制約」を局所化し、分析系はモダンなLinux/Cloud/Docker環境で構築するため。

## 4. 各レイヤーの詳細設計

### 4.1 Research Layer (データ取得・特徴量)

- **生データから特徴量へ**: J-Quantsの日足OHLCV、財務データから、リターン（1,5,20日）、移動平均乖離、PER、PBR、流動性（売買代金）などのファクターを抽出する。
- **スクリーニング**: PUSH登録上限（最大50銘柄）の制約を考慮し、前日夜間のうちに本番で監視する銘柄（20〜40銘柄）を絞り込む。極端な低位株や整理銘柄は事前除外する。

### 4.2 AI Analysis Layer (AI解析)

- **センチメント分析**: Yahoo!ニュースや日経等のテキストから、自然言語処理によりポジティブ・ネガティブを判定（`news_score`）。
- **マクロ・レジーム判定**: 重要イベント（日銀・FRB等）の抽出や、価格推移・VIXから現在の市場状態を Bull / Neutral / Bear に分類（`regime_score`）。

### 4.3 Strategy Layer (戦略・シグナル生成)

- **スコアの統合**: 定量ファクターとAIスコアを掛け合わせる。
  ```text
  final_score = 0.4*momentum + 0.2*value + 0.15*volatility + 0.15*liquidity + 0.10*news_score
  ```
- **レジーム別振る舞い**: Bear状態と判定された場合は、新規買いシグナルを抑制し、キャッシュ比率を高める。
- **出力形式の標準化**: json形式等でシグナル（対象銘柄、方向、許容リスク幅、有効期限等）を発行する。

### 4.4 Execution Layer (執行層)

- **発注制御**: kabuステーションAPIの制約（発注5件/秒 等）を守るOrder Queueを実装。
- **冪等性チェック**: 二重発注を防ぐための一意識別子（`client_order_id`）管理。
- **未約定・部分約定管理**: PUSH通信（WebSocket）による約定イベント処理と、ポジションDBのリアルタイム更新。

## 5. リスク管理アーキテクチャ（3段階ガード）

勝てるAI戦略を作る以前に、まずは**「壊れない（資金を飛ばさない）運用基盤」**を作ることが絶対条件である。

#### ガード1: シグナル前（ポートフォリオレベル）

- AIレジーム判定による相場急落時（Bear）の新規発注自動停止
- 連敗数閾値オーバー、または日次・週次ドローダウン（損失）上限超過での戦略一時停止

#### ガード2: 発注前（システム・執行レベル）

- 発注前の口座取引余力チェック
- 1銘柄への集中投資上限（例：総資産の10%）確認
- 同一銘柄への重複注文（二重発注）防止

#### ガード3: 発注後（監視・イレギュラーレベル）

- 注文後一定時間未約定の注文追跡・取り消し（想定外のジャンプアップ防止）
- 約定照合の不一致（DBと証券口座のズレ）検知時の**即時キルスイッチ発動**
- kabuステーションAPI応答異常時・WebSocket切断時の安全停止

## 7. 最終提言とフェーズ別導入

個人が作れる最強の日本株自動売買アーキテクチャは、**精巧なAIモデルを作ることではなく、安全な執行基盤を作ることがスタートライン**である。

### 導入ステップ

1. **Phase 1 (安全な最小構成)**: J-Quantsからのデータ取得・スクリーニング・バックテスト基盤の確立（AIなし・手動発注）。
2. **Phase 2 (半自動)**: ルールベース（モメンタム＋バリュー）のシグナル自動生成と、人手承認後のAPI注文送信。
3. **Phase 3 (フル自動)**: 3段階ガードとキルスイッチを有効にした小額完全自動発注。
4. **Phase 4 (AI高度化 - 最強構成)**: ニュースNLPモデルやレジーム判定AIを別サーバで稼働させ、シグナルに統合。複数戦略の同時運用。

**「強い戦略」は後からいくらでも改善できるが、壊れた執行系は一度で大きな損失（退場）につながる。**
したがって、AIという高度な武器を使うからこそ、本アーキテクチャが示す「厳格なリスク統制」と「環境分離」を何よりも優先して設計・実装すること。
