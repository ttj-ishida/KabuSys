- 文書名: 自動売買アーキテクチャ
- 対象: kabuステーション API と J-Quants Standard プランを用いた日本株自動売買基盤
- 作成日: 2026-03-15
- 版数: v1.0 (AI統合完全版)
- 関連文書: システム構築の要件定義\_AI統合完全版.md, 売買戦略仕様\_AI統合完全版.md

## 1. 本書の目的

本書は、三菱UFJ eスマート証券の **kabuステーション API** を利用した日本株自動売買システムについて、ニュースセンチメントやレジーム判定などのAI機能を組み込みつつ、**実運用を前提としたアーキテクチャ** を定義するものである。

本アーキテクチャでは、以下を重視する。

- 売買ロジック（AI・スコアリング）と執行ロジックの完全分離
- 研究系・分析系・戦略系・執行系・監視系の5層アーキテクチャモデルの採用
- 障害時（AI異常時含む）に安全側へ倒れるフェイルセーフ設計
- 監査可能性、再現性、保守性の担保

---

## 2. アーキテクチャ全体像（5層構造）

AI層を独立させ、最終売買判断はリスク管理層を経由する堅牢な構成とする。

```text
┌──────────────────────────────────────────────┐
│  1. 研究・分析系（Research Layer）           │
├──────────────────────────────────────────────┤
│ J-Quants API (Daily/Info/Financials)         │
│                 ↓                            │
│ ETL / Market Data Store / Feature Store      │
│                 ↓                            │
│ Screening Engine / Backtest Engine           │
└──────────────────────────────────────────────┘
                    ↓ (前日の日足ベース基礎スコア)
┌──────────────────────────────────────────────┐
│  2. AI判断モデル層（AI Analysis Layer）      │
├──────────────────────────────────────────────┤
│ News Collector (Yahoo!/日経等)                │
│                 ↓                            │
│ NLP / Sentiment Analyzer                     │
│ Macro Event & Market Regime Detector         │
│                 ↓                            │
│ AI Score Engine (news_score, regime_score)   │
└──────────────────────────────────────────────┘
                    ↓ (AI補助スコア・レジーム判定)
┌──────────────────────────────────────────────┐
│  3. 戦略系（Strategy Layer）                 │
├──────────────────────────────────────────────┤
│ Factor Model (Momentum/Value/Volatility)     │
│ Signal Generator (各スコアの最終統合)        │
│ Portfolio / Position Size Manager            │
│ Risk Manager (システムリスク・ポートフォリオリスク)│
└──────────────────────────────────────────────┘
                    ↓ (発注承認済みシグナル)
┌──────────────────────────────────────────────┐
│  4. 執行系（Execution Layer）                │
├──────────────────────────────────────────────┤
│ Order Manager (キュー・重複防止等)            │
│ Execution Engine (REST/WebSocket送受信)      │
│ Position / Fill (約定) Manager               │
│                 ↕                            │
│ kabuステーション API                           │
└──────────────────────────────────────────────┘
                    ↓ (非同期ストリーム・ステータス)
┌──────────────────────────────────────────────┐
│  5. 監視・運用統制系（Monitoring Layer）     │
├──────────────────────────────────────────────┤
│ Logging / Audit Trail Recorder               │
│ Alert Notifier (Slack / Mail)                │
│ Health Check / Scheduler / Session Control   │
│ 運用者手動キルスイッチ                         │
└──────────────────────────────────────────────┘
```

## 3. 設計方針とAIの役割

### 3.1 分離の原則

本番事故を防ぐため、以下を厳格に分離する。

- データの確定プロセス（日次処理）とリアルタイム処理（日中処理）
- AI等による「売買判断」と、証券APIへの「発注制御・状態管理」

### 3.2 AIの役割とフェイルセーフ

AIモデル（センチメント分析やマクロ解析）は **直接の発注意思決定を行わず、補助シグナル（スコアの加減算・レジーム判定によるポジション調整）として利用する。**
ニュースAPIの障害やAI推論サーバの遅延発生時は、AIスコアを無効（ゼロ）とし、通常の財務・テクニカルルールのベースライン戦略だけで稼働を継続できるフェイルセーフ設計とする。

## 4. 論理アーキテクチャ（各層の詳細）

### 4.1 研究・分析系 (Research / Data Layer)

- **ETL / Data Adapter**: J-Quantsからの夜間データ取得。差分取得や分割・併合の補正を行う。
- **Feature Store**: 再現性を担保するため、モメンタムやボラティリティなどの特徴量履歴をスナップショット保存する。
- **Screening Engine**: 翌営業日の監視対象リスト（Top N銘柄）を抽出し、kabuステーションPUSH登録数の制限（50件）に収める。
- **Backtest Engine**: 手数料・スリッページをモデル化し、Walk-forward検証を行う。

### 4.2 AI判断モデル層 (AI Layer)

- **News Collector / NLP Analyzer**: 定期的にニューステキストを取得し、BERT等LLM技術を用いてセンチメント（ポジ/ネガ/中立）およびトピックを判定する。
- **Regime Detector**: TOPIX等のマクロ指標推移やVIXから、市場状態をBull/Neutral/Bearの3分類に判定する。

### 4.3 戦略系 (Strategy / Risk Layer)

- **Signal Generator**: `final_score = 特徴量ベーススコア + news_score (最大10%)` の計算を行い、設定閾値到達でシグナル（買い/売り）を発火する。
- **Risk Manager**: シグナル受付後、**必ず執行前に以下のストッパーを通す**。
  - レベル1（システム）: API切断時や時刻同期異常時は棄却
  - レベル2（執行）: 二重発注、余力超過、時間帯外発注は棄却
  - レベル3（資金）: 最大保有数オーバー、ドローダウン超過時は棄却

### 4.4 執行系 (Execution Layer)

- **Execution Gateway (kabu ST Adapter)**: kabuステーション API の REST(発注/照会) および PUSH WebSocket(時価/約定) を隠蔽し、内部向けにイベント駆動で振る舞う。
- **Order Manager**: 冪等キー(`order_request_id`)を用いて二重発注を防ぐ。通信エラー等の「状態不明」時は直ちに発注キューを停止し、照会APIでの状態確定を優先する。

### 4.5 監視・運用統制系 (Monitoring / Audit Layer)

- **Session Controller**: 寄付前、前場、昼休み、後場、引け処理のタイムスケジュールを管理する。
- **Audit Recorder**: 単なるログではなく、「なぜそのシグナルが出たか（スコア群）」と「なぜ発注された/棄却されたか」をDBに永続化し監査可能にする。

## 5. 推奨環境アーキテクチャ

本番稼働時のリスク分散のため、システムを論理・物理的に分割配置する構成を推奨する（PoC段階は単一PC可）。

```text
[ 分析・AI サーバー群 (Linux/Cloud) ]
 ├─ J-Quants & News ETL
 ├─ PostgreSQL (Data Warehouse)
 ├─ AI 推論コンテナ (Python, NLP models)
 └─ Screening / Backtest Engine

[ 執行PC (Windows ローカル端末) ]
 ├─ kabuステーション アプリ
 ├─ Execution Node (Pythonプロセス)
 │   ├─ Strategy / Risk Manager
 │   ├─ Order Manager
 │   └─ Monitoring Agent (PCリソース/通信監視)
```

この分離構成により、重いAI処理や大量データ処理で執行用Windows PCのCPU領域・メモリを圧迫し、UIやAPIのフリーズにつながるリスクを排除する。

## 6. データモデル方針と保存ポリシー

### 6.1 主なテーブル構造

- `symbols` / `trading_calendar`: 基礎マスタ
- `market_data_daily` / `financials`: J-Quants 生データ
- `feature_snapshot` / `ai_sentiment_log`: 分析結果スナップショット
- `signal_events`: シグナル発生履歴（根拠スコア含む）
- `order_requests` / `executions`: 発注・約定・取消履歴（監査用）
- `positions` / `account_summary`: ポジション・余力履歴

### 6.2 冪等・トレーサビリティ設計

すべての注文は以下のキー цепочка で一貫してトレースできるように設計する。
`business_date` -> `strategy_id` -> `signal_id` -> `order_request_id` -> `broker_order_id` (証券会社受付ID)

## 7. 障害検知・フェイルオーバーアーキテクチャ

### 7.1 対応方針

基本原則として **自動売買システムにおける「状態の不確定」は最大のリスク** とする。

- 注文後、タイムアウトなどで約定状態が不明な場合は、状態が確定するまで**以降の新規シグナルを全無視**する。
- kabuステーションアプリ自体のハングアップやWebSocket PUSH切断を検知した場合、システムを**セーフモード（新規建て不可・手仕舞いのみ可、または全停止）**へ移行させる。

### 7.2 再起動・自動復旧の標準フロー

意図せぬ再起動やクラッシュからの復帰時、以下のシーケンスを強要する。

1. kabuステーション稼働およびAPIトークン有効性確認
2. APIによる「現在の全保有ポジション」と「内部DBポジション」の照合（不一致時は手動介入待ち）
3. 未約定の有効注文の棚卸しとキュー再構築
4. 監視銘柄（最大50）のPUSH再登録
5. 安全確定後、自動売買ステータスを稼働にセット

## 8. 開発・導入フェーズ

### Phase 1: 研究・基礎基盤 (Research & Backtest)

- J-Quants ETL構築、スクリーニング実装
- 基本特徴量とバックテストエンジンの構築・整合性検証

### Phase 2: ルールベース本番接続 (Execution PoC)

- 執行層・リスク層の構築と、kabuステーション検証環境での疎通
- AIなしのモメンタム・バリュースコアによる小額実売買取組開始

### Phase 3: AIモデルの統合 (AI Layer Integration)

- ニュースデータ収集とNLPセンチメントモデルの実装
- レジーム判定ロジックの実装と、スコア計算式への統合

### Phase 4: 高度化と監視システム (Ops / Advanced)

- メトリクス監視（Grafana等）やアラート通知の高度化
- セクター別AIモデル、複数戦略並列運用への拡張
