
# DataPlatform.md

## 1. 目的

本ドキュメントは、日本株自動売買システムにおける **データ基盤（Data Platform）** の設計を定義する。

本システムは完全なデータ駆動型システムであり、以下の処理がすべてデータ基盤上で行われる。

- 市場データ収集
- ニュースデータ収集
- 特徴量生成
- AIスコア生成
- 売買シグナル生成
- バックテスト
- 実運用ログ保存

そのため、**安定したデータ取得・保存・管理の仕組み**がシステムの中核となる。

---

# 2. データ基盤の基本思想

本データ基盤は以下の原則で設計する。

1. **全データを履歴として保存する**
2. **バックテスト再現性を保証する**
3. **データ更新と分析処理を分離する**
4. **時系列整合性（look-ahead防止）を徹底する**
5. **将来のデータ拡張に対応可能な構造にする**

# 4. データレイヤー構造

データは以下の3層構造で管理する。

```
Raw Layer
↓
Processed Layer
↓
Feature Layer
```

---

# 5. Raw Layer（生データ）

外部ソースから取得したデータをそのまま保存する。

目的

- 元データの保存
- 再処理可能な状態の保持

例

| テーブル | 内容 |
|--------|------|
raw_prices | 株価 |
raw_financials | 財務 |
raw_news | ニュース |
raw_fx | 為替 |

---

# 6. Processed Layer（整形データ）

Rawデータを分析可能な形式に整形する。

処理

- 欠損補完
- 重複除去
- 日付整形
- 銘柄コード統一

例

| テーブル | 内容 |
|--------|------|
prices_daily | 日足 |
financials_clean | 財務 |
news_clean | 整形ニュース |

---

# 7. Feature Layer（特徴量）

分析やAIモデルで利用する特徴量を保存する。

例

| テーブル | 内容 |
|--------|------|
factor_momentum | モメンタム |
factor_value | バリュー |
factor_volatility | ボラティリティ |
news_sentiment | ニューススコア |
market_regime | 市場レジーム |

---

# 8. データ更新パイプライン

データ更新はETLパイプラインで実行する。

```
Data Fetch
↓
Raw Storage
↓
Data Cleaning
↓
Feature Generation
↓
Signal Calculation
```

更新タイミング

| 処理 | タイミング |
|----|----|
市場データ取得 | 夜間 |
ニュース取得 | 定期 |
特徴量生成 | 夜間 |
AI分析 | 夜間 |
戦略スコア生成 | 夜間 |

# 11. データ品質管理

チェック項目

- 欠損データ
- 重複データ
- 異常値
- 日付不整合

異常検知時

```
ETL停止
```
または

```
警告ログ出力
```

---

# 12. データ保存方針

保存対象

- 市場データ
- ニュースデータ
- AIスコア
- 戦略シグナル
- 売買履歴

保存期間

```
永続保存
```

# 13. バックテスト用データ

- AIスコア履歴
- 売買履歴

```
未来データ参照禁止
```

# 16. まとめ

データ基盤は以下の役割を担う。

```
データ取得
↓
データ保存
↓
特徴量生成
↓
AI分析
↓
戦略計算
```

本データ基盤により、バックテストと本番運用の双方で
**再現性の高い分析基盤**を実現する。
