# Data Platform (データ・インフラ基盤)

- 対象: 日本株自動売買基盤における全データパイプライン
- 版数: v1.0

---

## 1. 目的とデータ駆動の原則

本システムは、精巧な推測・予測（アルゴリズム）の前に、**「完全で一貫性のあるデータ」**が全てを決定する「データ駆動（Data-Driven）」システムである。

日中・夜間問わず大量に発生する、J-Quantsの市場データ、ニューステキスト、AI推論の特徴量、そして日々のバックテスト結果や本番の取引履歴を確実に正規化・安全にストアする基盤を定義する。

---

## 2. データソースとETLパイプライン

### 2.1 主なデータソース

1. **J-Quants API (Standard プラン)**
   - 銘柄一覧・上場情報
   - 日足株価（OHLCV）
   - 財務諸表（四半期ごとのBS/PLサマリ）
   - コーポレートアクション（株式分割・併合など）
2. **外部ニュース・テキストソース**
   - Yahoo!ニュース、日経ヴェリタス 等のRSSやスクレイピングデータ
   - （将来的には）日銀・海外中央銀行のマクロ経済発表テキスト
3. **kabuステーション API (自システム生成データ)**
   - 発注要求ログ
   - WebSocketからのPUSH約定データ
   - 口座評価損益、ポジション現況

### 2.2 ETL (Extract, Transform, Load) 方針

- **Idempotency (冪等性)**: データ取得・変換ジョブは、何度再実行しても同じ結果（重複排除）となるように設計する。
- **Raw/Processed 分離**: 取得したAPIのレスポンス（生JSON等）は「Raw Store」に捨てずに保存し、その後にパース・加工した状態を「Processed Store / RDB」へ保存する（パースエラー時の再処理担保）。
- **差分更新とスロットリング**: J-Quants APIの利用上限（120リクエスト/分）を超えないようにスロットル制御を入れ、前日からの差分のみを効率よく取得する。

## 3. データベース構造と保存ポリシー

本システムでは、RDB（PostgreSQL推奨）をメインストアとし、用途に応じてNoSQL（Redis等）をキャッシュ・メッセージキューとして併用する。

### 3.1 主なテーブル構造の概念

| ドメイン | テーブル群 (例) | 概要 |
|---|---|---|
| **Master** | `symbols`, `trading_calendar` | 銘柄マスタ、上場廃止フラグ、取引日カレンダー |
| **Market** | `market_data_daily`, `financials` | J-Quantsからの日足、発行済株式数、財務指標 |
| **Feature** | `feature_snapshot`, `ai_sentiment_log` | 日次で計算されたモメンタムやボラティリティ、NLPスコア |
| **Strategy** | `signal_events` | いつ、どの銘柄に、何のスコア根拠でシグナルが出たか |
| **Execution** | `order_requests`, `executions` | 証券会社への発注履歴、約定履歴、キャンセル履歴 |
| **Portfolio** | `positions`, `account_summary` | その時点での内部保有株、平均取得単価、評価損益 |

### 3.2 冪等・トレーサビリティ設計

システム上で「シグナル」から「約定」に至るフローを完全にトレースするため、全テーブルを一貫したキー（`uuid`等の識別子）で連結する。

```text
business_date (営業日)
  └─ strategy_id (戦略バージョン)
       └─ signal_id (シグナル固有ID)
            └─ order_request_id (内部発注ID / 冪等キー)
                 └─ broker_order_id (証券会社受付ID)
```

監査ログ（Audit Trail）として、エラーで行われなかった発注や、リスク管理マネージャによって棄却されたシグナルも、ステータス（例: `REJECTED_BY_RISK`）を変えて必ず永続化する。

## 4. 基礎データの補正ルール

本番執行基盤とバックテスト環境の乖離（ルックアヘッド・バイアス）を生じさせないため、データプラットフォーム側で以下の厳密な管理を行う。

1. **財務データの発生日**: 
   - 財務発表日（開示日）を基準に「いつからシステムがそのデータを知り得たか」をフラグ管理する。15時以降の発表は、翌営業日扱いとする。
2. **価格の調整（分割・併合）**:
   - 株式分割が実施された場合、過去の株価情報に対して「調整済み価格 (Adjusted Price)」を再計算して一元提供する。
   - `調整前` と `調整後` が混在しないビューを提供する。
