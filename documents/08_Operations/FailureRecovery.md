# FailureRecovery.md

## 1. 目的

本ドキュメントは、日本株自動売買システムにおける **障害対応（Failure
Recovery）** 手順を定義する。

目的:

-   障害発生時の迅速な対応
-   自動売買の安全停止
-   データ整合性の維持
-   システム復旧手順の標準化

対象環境:

-   Single Windows PC
-   kabuステーションAPI
-   Python自動売買システム

------------------------------------------------------------------------

# 2. 障害分類

障害は以下のカテゴリに分類する。

  Category             内容
  -------------------- -----------------
  System Failure       PC停止・OS障害
  API Failure          証券API接続障害
  Execution Failure    注文送信エラー
  Data Failure         データ取得失敗
  Strategy Failure     戦略計算エラー
  Monitoring Failure   監視停止

------------------------------------------------------------------------

# 3. Kill Switch

重大障害時は **自動売買を停止する。**

Kill Switch 条件:

-   最大ドローダウン超過
-   API接続断
-   Execution Engine停止
-   異常注文検知

実行処理:

    1. 新規注文停止
    2. Execution Engine停止
    3. アラート送信
    4. 手動確認

------------------------------------------------------------------------

# 4. API障害

## 症状

-   kabuステーションAPI接続不可
-   注文送信失敗

## 対応

    1. 再接続試行
    2. retry_attempts回リトライ
    3. 失敗時はKill Switch

------------------------------------------------------------------------

# 5. PC停止

## 症状

-   Windows再起動
-   システム停止

## 復旧手順

    1. PC起動
    2. DB整合性チェック
    3. ポジション確認
    4. Execution再起動

------------------------------------------------------------------------

# 6. Signal Queue破損

## 症状

-   signal_queue読み込みエラー

## 対応

    1. signals再生成
    2. portfolio再計算
    3. signal_queue再構築

------------------------------------------------------------------------

# 7. 注文エラー

## 症状

-   order rejected

## 対応

    1. 注文ログ確認
    2. リトライ
    3. エラー通知

------------------------------------------------------------------------

# 8. データ取得失敗

## 症状

-   J-Quants API failure

## 対応

    1. 再取得
    2. 前日データ利用
    3. Night Batch停止

------------------------------------------------------------------------

# 9. ポジション不整合

## 症状

-   DBポジションと証券口座不一致

## 対応

    1. 口座残高取得
    2. DB修正
    3. ポートフォリオ再構築

------------------------------------------------------------------------

# 10. Monitoring停止

## 症状

-   監視プロセス停止

## 対応

    1. monitoring_service再起動
    2. ログ確認

------------------------------------------------------------------------

# 11. 復旧確認

復旧後は以下を確認する。

-   ポジション
-   未処理注文
-   シグナルキュー
-   API接続

------------------------------------------------------------------------

# 12. まとめ

Failure Recovery設計の原則

-   Fail Safe
-   Kill Switch
-   Data Integrity
-   Manual Override

この設計により **安全な自動売買運用を実現する。**
