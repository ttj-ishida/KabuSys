バグ
1) URLError/OSError リトライ時に待機がない\n- 対象: _request 内 except (urllib.error.URLError, OSError) ブロック
- 期待: HTTPError系と同様に指数バックオフで sleep すべき\n- 現状: wait を計算・ログ出力はしているが time.sleep(wait) がない\n- 影響: ネットワーク不安定時に即時に連打し、回復待ちができない\n- 修正案:
  except (urllib.error.URLError, OSError) as e:
      last_exc = e
      if attempt < _MAX_RETRIES - 1:
          wait = _RETRY_BACKOFF_BASE ** attempt
          logger.warning("Network error ...")
          time.sleep(wait)  # これが必要\n      continue  # 明示しておくと読みやすい

2) get_id_token が 401 → _request が自動リフレッシュ → 再度 get_id_token ... の無限再帰の可能性
- 対象: get_id_token() が _request("/token/auth_refresh") を呼ぶが、_request 側で401時に get_id_token() を呼ぶ設計
- 影響: refresh_token が失効/不正な場合などに無限再帰 → スタックオーバーフロー/DoS の恐れ
- 修正案: _request に allow_refresh フラグを追加（デフォルト True）。tokenリフレッシュAPI呼び出し時は False を渡す。
  def _request(..., allow_refresh: bool = True) -> Any:
      ...
      if status == 401 and allow_refresh and not _token_refreshed:
          token = get_id_token()
  get_id_token() 側:
      data = _request("/token/auth_refresh", method="POST", json_body={...}, allow_refresh=False)

3) ドキュメンテーションと挙動の不整合
- _request の docstring は「最大リトライ超過時に RuntimeError」とあるが、再試行対象外のHTTPエラー(例: 400/403/404など)はそのまま再raiseしている。
- 提案: docstring を「再試行対象外のHTTPエラーは urllib.error.HTTPError を送出する。再試行枯渇時は RuntimeError を送出する」に更新。
