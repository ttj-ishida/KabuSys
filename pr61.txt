## AI Code Review

以下、観点ごとにレビューします。結論から言うと、全体の方向性はよいですが、実運用での安定性に関わるバグ・設計課題がいくつかあります。特に認証と HTTP メソッド、カレンダーの区分解釈、DuckDB への UPSERT 互換性は修正を推奨します。

バグ
- 認証メソッド/ドキュメント不整合
  - get_id_token は _request("/token/auth_refresh", params=…) を呼んでいますが、現行の J-Quants v1 API は POST で refreshtoken を JSON ボディに載せる想定です（GET クエリで動かない可能性が高い）。
  - _request は GET 固定の実装で、POST が打てない。→ 認証は失敗し得ます。
  - 対応案: _request に method と data を受け取り、POST を実装するか、専用の _post_request を追加してください。
- id_token の自動取得の未実装
  - _request の docstring に「id_token 省略時は refresh_token から自動取得」とありますが、実装されていません。→ ドキュメントと実装が食い違い。
  - 対応案: id_token が None の場合 settings.jquants_refresh_token から get_id_token() を呼ぶロジックを実装するか、docstring を修正する。
- 401 対応の欠如
  - id_token の有効期限切れ時（HTTP 401）に refresh_token からの再取得＋リトライが無い。→ 長期運用で死にやすい。
  - 対応案: HTTP 401 を受けたら 1 回だけ id_token をリフレッシュしてリトライ。
- マーケットカレンダーの区分解釈の可能な誤り\n  - is_trading_day を HolidayDivision == "0" のみ True にしていますが、半日("3")、SQ("2") は通常「取引あり」の日です。現状だと半日営業日や SQ 日が非営業日として扱われます。
  - 対応案: is_trading_day = HolidayDivision in {"0","2","3"}（定義は要ドキュメント確認）。is_half_day, is_sq_day の付与は現状のままで OK。
- DuckDB の INSERT OR REPLACE 互換性リスク
  - DuckDB のバージョンによっては INSERT OR REPLACE のサポート／挙動が異なることがあります（SQLite 互換のつもりで書くと刺さることがある）。運用環境の DuckDB バージョンで互換を要確認。
  - 安全策: MERGE INTO もしくは INSERT … ON CONFLICT DO UPDATE を使う（DuckDB 0.9+）。MERGE のほうが無難。
- データ型の不整合の可能性
  - Volume（出来高）や TurnoverValue（売買代金）は本来整数です。_to_float で float にして保存すると、DuckDB 側が BIGINT の場合に型不一致や不要なキャストが発生します。
  - 対応案: _to_int を用意して整数フィールドは int に。
- 小さな不具合/不整合
  - _RateLimiter の docstring が「トークンバケット方式」とありますが、実装は固定間隔のスロットリングです（トークンバケットではない）。→ ドキュメント修正か実装変更。
  - get_id_token 内の url 変数が未使用。
  - 未使用 import: Path, timedelta を import していますが未使用。

セキュリティ
- トークンのログ混入は現状回避されており良いです。
- 攻撃耐性
  - 429/5xx のリトライは OK。ただし 408（Request Timeout）等の一部 4xx もネットワーク越しでは再試行が有効な場合があり、再試行対象に含めると堅牢性が上がります。

可読性

設計

## AI Code Review

総評\n- 方向性は良いです。API クライアントとしての基本（レート制限、リトライ、401 時のリフレッシュ、ページネーション、DuckDB への冪等保存）を押さえています。

指摘（バグ）\n- market_calendar の HolidayDivision の比較\n  - r.get("HolidayDivision") が int の場合があり得ますが、現在は文字列リテラル "0"/"2"/"3" と比較しています。型不一致で全て False になり、is_trading_day 等が誤る可能性があります。
  - 対応案:
    - dv = str(r.get("HolidayDivision", "")) としてから比較する
    - あるいは数値比較に統一する
- 401 リフレッシュの「1回だけ」保証が実装されていない\n  - コメントでは「401 受信時はトークンを自動リフレッシュして 1 回リトライ」とありますが、コード上は attempt ごとに毎回リフレッシュが行われる可能性があります。無駄なリフレッシュや、トークン側のレート制限を踏む恐れがあります。
  - 対応案: refreshed = False フラグを導入し、401 でのリフレッシュは 1 回のみに制限する。
- JSON デコードエラー時のメッセージが不十分
  - API が HTML/テキストを返した場合に json.loads が JSONDecodeError を投げますが、現在は捕捉されず、原因調査が難しいです。
  - 対応案: JSONDecodeError を捕捉し、HTTP ステータス・先頭 N バイトのレスポンスボディをログに出してから RuntimeError で包んで投げる。
- save_financial_statements のキー設計の妥当性
  - ON CONFLICT (code, report_date, period_type) において period_type に TypeOfDocument を使っていますが、J-Quants の statements は「四半期種別（期/四半期）」や改訂/訂正の別、同一開示日の差分などを区別するフィールドが別にあります。TypeOfDocument だけだと衝突や上書きミスの可能性が高いです。
  - 最低限、ユニーク性を満たすために code + DisclosedDate + TypeOfCurrentPeriod（あるいは CurrentPeriodEndDate）+ AmendmentFlag 等の組み合わせを検討してください。J-Quants の仕様に沿ってキー設計を見直すことを強く推奨します。
- fetched_at の型整合性
  - fetched_at を ISO 文字列（+00:00 付き）で挿入しています。DuckDB 側の列型が TIMESTAMP（タイムゾーンなし）の場合、パースに失敗するか、意図せぬ変換になります。TIMESTAMPTZ を使うか、UTC naive に正規化してから文字列化するなど、DB 列型と合わせてください。
- save_* 系で不完全なレコードの混入
  - 例えば code を str(r.get("Code", "")) として空文字を許しています。PRIMARY KEY に code を含むなら空文字は実質的に 1 銘柄として衝突します。不完全レコードは除外するか、必須フィールドが欠損しているデータはスキップ/ログ出力してください。

セキュリティ
- Authorization ヘッダやトークンをログに出していないのは良いです。
- HTTP タイムアウトは設定済み（timeout=30）。ただし、DNS ハングや一部経路で長時間待たされる可能性はあるため、再試行の最大所要時間を設計決定しておくと安心です（総合タイムアウト）。
- User-Agent を明示的に付けることを推奨します（どのクライアントかの識別、先方トラブルシュートに有用）。
- リトライ時にレートリミットと指数バックオフが二重に効く形になっているため、攻撃的にはならない一方で遅延過多の可能性があります。最大待機時間・ジッタの導入などで平準化を。

設計
