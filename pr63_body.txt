## AI Code Review

総評
- テスト対象が明確で、モックにより外部依存を遮断しつつ、ETL の冪等性・スキーマ整合性・品質チェックまで網羅しており、とても良い構成です。
- 一方で、未使用インポートや私的関数への強い結合、将来の実装変更に対して脆いアサーションなど、いくつか改善余地があります。

バグ/不具合の可能性
- 未使用インポート
  - time, urllib.error, urllib.request, data_schema が未使用です。テスト環境のリンタで失敗/ノイズになります。削除を推奨します。
- DuckDB バージョン依存のリスク
  - ON CONFLICT のサポートは DuckDB のある程度新しいバージョンが必要です。コメントで FK の問題に触れていますが、ON CONFLICT 自体も下限バージョン明記や pyproject の制約と整合しているか確認を推奨します。
- 無限ループ検知テストの厳しさ
  - test_fetch_daily_quotes_dedup_pagination_key は call_count == 2 を前提にしています。実装が「同一キーを受け取った時点で break」なのか「次ページ取得後に同一キーを検知して break」なのかに依存し、実装が妥当でも失敗する可能性があります。回数の上限を設けたアサーション（例: 2 回以下かつ 1 回以上）や、同一キーを見たら抜けること自体を検証する形に緩和を検討ください。
- データ不整合の境界ケース未検証
  - save_financial_statements の主キー欠損スキップや ON CONFLICT 更新パスの検証がありません（raw_prices と同等の検証を追加推奨）。
  - market_calendar で未知の HolidayDivision（例: "9"）に対するデフォルト動作が未検証です（未知値で is_trading_day=False 等の期待仕様があればテスト追加推奨）。
  - save_daily_quotes の Code が空文字列はスキップしていますが、空白のみやゼロパディング/非数値（例: "  ", "ABC"）の扱いを仕様化・テスト化すると堅牢になります。

セキュリティ
- ネットワークアクセスは全面的にモックされており安全です。
- 環境変数の取り扱いは monkeypatch でスコープ内に閉じられており良いです。
- 懸念は特にありません。

可読性
- テスト名・日本語 Docstring が明確で意図が伝わります。
- ヘルパー生成関数でサンプルが統一されており、可読性良好です。
- 改善提案
  - 重複パターン（複数ページ、複数銘柄など）は pytest.mark.parametrize で簡潔にできます。
  - market_calendar の区分→フラグ変換テストはパラメタライズで意図がさらに明快になります。

設計
- モックの粒度
  - jquants._request（私的関数）を直接モックしており、実装詳細への結合が強めです。将来 _request 名称/配置変更で壊れます。HTTP 層を注入可能にする、あるいは公開 API（fetch_*）が依拠するインタフェースを公開してそこをモックする設計にすると保守性が上がります。
- スキーマ定義の重複
  - _MINIMAL_DDL をテストに直書きすると本番スキーマとのドリフト懸念があります。schema.py 側に「テスト用に制約を緩和した DDL」を用意する、あるいは init_schema のオプションで FK/カスケードを抑止できるようにして、単一ソースから生成するのが望ましいです。

パフォーマンス
- すべてインメモリで件数も小さく、テスト時間・リソース消費とも十分小さいです。懸念なし。

追加であると良いテスト
- fetch_* 系
  - フィルタ引数（銘柄コード/日付範囲など）を渡した際、_request に正しい params が渡るかの検証。
  - 3 ページ以上のページネーション、または最後が空配列のケース。
  - エラーハンドリング（429/5xx/ネットワーク例外）や再試行ロジックがある場合、その分岐の検証。
- save_* 系
  - raw_financials の ON CONFLICT DO UPDATE（再保存で値が更新される）検証。
  - market_calendar の上書き（既存行の holiday_name などが更新される）検証。
  - NULL や欠損（非 PK カラム）が混在する場合の受け入れ確認。
- quality 連携
  - しきい値（例: spike の 50%）を定数から参照する形にして、値変更に強いテストへ。もしくは許容比率を引数化してテスト。
  - run_all_checks が複数の問題を同時に検出するシナリオ（例: 欠損かつスパイク）での安定性。

細かな改善（ニト）
- 数値比較は DECIMAL→float の変換で丸め誤差が入る可能性があるため、Decimal での比較か、pytest.approx での比較を検討。
- fixture mem_db は現状でも安全ですが、将来的に並列実行を増やす場合は関数スコープ明記（scope="function"）や一意の in-memory DB 名を用いることで衝突リスクをさらに下げられます。
- import の整理（未使用削除）。また、tests 側から本番コードにのみ存在する import（data_schema）は、使用しないなら削除し、必要ならテスト名/コメントで意図を補足。

まとめ
- 現状でも有用な回帰防止網を提供できています。上記の未使用インポート削除、ページネーション重複キーのアサーション緩和、私的関数への依存度低減、いくつかの境界ケース追加を行うと、将来の変更にもより強いテストスイートになります。
